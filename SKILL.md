# Rich Table — Agent Reference

`rich-table` is an Obsidian plugin that renders rich interactive tables inside
fenced code blocks. This skill covers everything an agent needs to create and
modify these tables in Markdown files.

> **Obsidian-only.** The `rich-table` block will not render in standard
> Markdown editors or GitHub previews.

---

## Block structure (v2 format)

````markdown
```rich-table
---
version: 2
title: Optional table title
columns:
  - id: c_000000
    name: Task
    width: 220
  - id: c_000001
    name: Status
    type: task-status
    width: 110
  - id: c_000002
    name: Owner
    width: 120
rows:
  - id: r_000000
    cells:
      c_000000: Design architecture
      c_000001: done
      c_000002: "[[Alice]]"
  - id: r_000001
    cells:
      c_000000: Build API
      c_000001: in-progress
      c_000002: "[[Bob]]"
merges:
  - anchor: r_000000.c_000000
    end: r_000001.c_000000
styles:
  - target: header
    bold: true
    bg: "#e8f0fe"
  - target: r_000000.c_000001
    bg: "#f0fdf4"
footer: "Updated weekly · click any cell to edit"
---
<!-- Generated by Rich Table. Do not edit below — data source is the YAML front-matter above. -->
| Task               | Status      | Owner     |
| ------------------ | ----------- | --------- |
| Design architecture | done       | [[Alice]] |
| Build API          | in-progress | [[Bob]]   |
```
````

**Structure rules:**
- The **YAML front-matter** is the only data source. The pipe table below `<!-- Generated … -->` is a read-only mirror regenerated on every write — do not edit it.
- `version: 2` is required for the current format.
- Columns and rows each have a stable `id` field. Use sequential IDs: `c_000000`, `c_000001`, … for columns and `r_000000`, `r_000001`, … for rows.
- Cell values are stored in `rows[].cells` as a record keyed by column ID.

## YAML fields

### `version` (number, required)
Always `2` for the current format.

### `title` (string, optional)
Displayed above the table. Supports inline Markdown.

### `columns` (array)
Each entry has:

| Field    | Type | Default | Description |
| -------- | ---- | ------- | ----------- |
| `id`     | string | — | Stable ID, e.g. `c_000000` |
| `name`   | string | — | Column header text |
| `type`   | string | — | Built-in or custom choice type (see below) |
| `width`  | number | auto | Column width in px |
| `align`  | `left` \| `center` \| `right` | — | Text alignment |
| `hidden` | boolean | false | Hidden (collapsed) column |
| `filter` | array of strings | — | Values to SHOW for this column (row filtering). Managed automatically by the plugin UI. |

```yaml
- id: c_000001
  name: 状态
  filter:
    - done
    - in-progress
```

> **Quote column names containing YAML special characters.**  
> Characters `[ ] { } : , #` need double-quoting:
> ```yaml
> - id: c_000000
>   name: "rx_buf[] type"   # ✅
> ```

### `rows` (array)
Each entry has:

| Field    | Type | Description |
| -------- | ---- | ----------- |
| `id`     | string | Stable ID, e.g. `r_000000` |
| `cells`  | object | Map of column-id → cell content string. Sparse — a missing colId key means an empty cell, never an error; the parser must not require every column to be present. |
| `hidden` | boolean | Row is hidden (collapsed) |
| `height` | number | Forced row height in px |

Omit keys for empty cells (sparse storage). Multi-line cell content uses YAML block scalar `|-`:

```yaml
  - id: r_000002
    cells:
      c_000000: |-
        - First item
        - Second item
        - Third item
      c_000001: todo
```

### `merges` (array, optional)
Each entry is a rectangle identified by anchor (top-left) and end (bottom-right) cells:

```yaml
merges:
  - anchor: r_000000.c_000000
    end:    r_000002.c_000000
  - anchor: r_000001.c_000002
    end:    r_000002.c_000003
```

The anchor cell holds the displayed content; covered cells should have empty or omitted values.

### `styles` (array, optional)
Each entry has a `target` and one or more style properties:

| Property | Type    | Description |
| -------- | ------- | ----------- |
| `target` | string  | Which cells to style (see target reference below) |
| `bg`     | string  | Background color (CSS color string) |
| `color`  | string  | Text color (CSS color string) |
| `size`   | number  | Font size in px |
| `bold`   | boolean | Bold text |
| `italic` | boolean | Italic text |

```yaml
styles:
  - target: header
    bold: true
    bg: "#e8f0fe"
  - target: c_000001
    bg: "#e6f4ea"
  - target: r_000000.c_000002
    color: "#c0392b"
    size: 14
```

### `footer` (string or array of strings, optional)
Displayed below the table. Supports inline Markdown. Use an array for multiple lines.

```yaml
footer: "Last updated 2025-01"
# or multi-line:
footer:
  - "* Estimates only"
  - "Source: internal data"
```

### `theme` (string, optional)
Built-in visual theme applied to the table. Themes control appearance only
(colors, borders, fonts) and never affect data.

| Value | Appearance |
|-------|-----------|
| *(omit)* | Default — no special styling |
| `academic` | Booktabs style: three horizontal rules only (toprule / midrule / bottomrule), no vertical lines, no backgrounds |
| `plain` | Purple/blue gradient header, rounded corners with outline border, row hover highlight, centered |

```yaml
theme: academic
```

### `locked` (boolean, optional)
When `true`, all graphical editing is disabled for this table.

### `collapsed` (boolean, optional)
When `true`, only the title (if any) and header row render — the table body
and footer are hidden. Toggle via the fold-icon button in the top-left corner.

### `sort` (object, optional)
Live/persistent row sort — the popup menu opened by selecting a single column
in the column selector strip has two sort modes:
- **"Sort ascending" / "Sort descending"** — one-time: reorders `rows[]` in the
  YAML immediately, then forgets about it. No `sort` field is written; manual
  drag-reorder keeps working normally afterward.
- **"Keep sorted ascending" / "Keep sorted descending" / "Clear live sort"** —
  live: writes this `sort` field instead of touching `rows[]`. Re-applied on
  every render (so newly-inserted rows also land in the right place), and
  disables the row drag-grip while active since the display order is derived,
  not stored. A live sort in effect also shows a small always-visible
  indicator directly in that column's header cell (same corner as the filter
  button, stacked above it if both are active on that column), independent of
  the popup menu, specifically so it's never silently forgotten — click it to
  clear.

Both modes are omitted/disabled while any `merges` entry spans more than one
row (filter stays on its own funnel icon in the header — unaffected by any of
this).

```yaml
sort:
  colId: c_000001
  dir: desc  # or asc
```

## Style target reference (v2)

| Target | Meaning |
|--------|---------|
| `header` | Entire header row |
| `header.c_xxx` | Single header cell (column `c_xxx`) |
| `r_xxx` | Entire data row |
| `c_xxx` | Entire column, including header |
| `r_aaa:r_bbb` | Row range |
| `c_aaa:c_bbb` | Column range |
| `r_aaa.c_aaa:r_bbb.c_bbb` | Rectangle range |
| `r_xxx.c_yyy` | Single data cell |

**Cascade priority** (lower wins first, higher overrides):
1. Whole row / whole column (`r_xxx`, `c_xxx`)
2. Row range / column range (`r_aaa:r_bbb`, `c_aaa:c_bbb`)
3. Rectangle range (`r_aaa.c_aaa:r_bbb.c_bbb`)
4. Single cell (`r_xxx.c_yyy`, `header.c_xxx`)

Per-attribute merge — a cell rule only overrides the properties it explicitly sets.

## Cell content

Cell values are plain strings rendered by Obsidian's `MarkdownRenderer`. Supported content:

| Content | Example |
|---------|---------|
| Plain text | `Hello world` |
| Bold / italic | `**bold** · *italic* · ***both***` |
| Highlight | `==highlight==` |
| Wikilink | `[[Note title]]` |
| External link | `[label](https://example.com)` |
| Image | `![](https://example.com/img.png)` |
| Local image | `![[image.png]]` |
| Inline math (KaTeX) | `$E=mc^2$` |
| Line break | `Line 1<br>Line 2` |
| List items | Use YAML block scalar `\|-` with `- ` prefix lines |

**List example** (the `- ` prefix lines render as a compact bulleted list):
```yaml
c_000000: |-
  - First item
  - Second item
  - Third item
```

**Math example** (backslash is literal in YAML):
```yaml
c_000000: '$\sqrt{2} \approx 1.414$'
```

## Built-in column types

| Type ID       | Values (use the value string in cells) |
| ------------- | -------------------------------------- |
| `task-status` | `todo` · `pending` · `in-progress` · `done` · `cancel` |
| `priority`    | `high` · `medium` · `low` |
| `boolean`     | `yes` · `no` |
| `rating`      | `1` ★ · `2` ★★ · `3` ★★★ · `4` ★★★★ · `5` ★★★★★ |
| `effort`      | `xs` · `s` · `m` · `l` · `xl` |
| `approval`    | `approved` · `pending` · `rejected` |
| `date`        | ISO date string `YYYY-MM-DD` |

## Common patterns

### Minimal table (no YAML needed)
For very simple tables, omit the YAML and use the legacy v1 format — the plugin will show an upgrade banner but still render the table:

````markdown
```rich-table
| Name   | Role      | Team  |
| ------ | --------- | ----- |
| Alice  | Engineer  | Core  |
| Bob    | Designer  | UX    |
```
````

For new tables, always prefer v2 format with explicit IDs.

## Themes

Set the `theme` field in the YAML front-matter to apply a built-in visual theme.

### Available themes

**`academic`** — LaTeX booktabs style
- Only three horizontal rules: thick line above header, medium line below header, thick line after last row
- No vertical lines, no cell backgrounds, minimal distraction

```yaml
theme: academic
```

**`plain`** — Colorful and animated
- Rainbow gradient header that slowly cycles hue, rounded corners with an animated
  outline border and comet-flash effect
- Row hover highlight with a cursor-following glow
- Table centered in the reading pane

```yaml
theme: plain
```

### Applying a theme

Add `theme:` to the YAML front-matter alongside other fields:

````markdown
```rich-table
---
version: 2
theme: academic
columns:
  - id: c_000000
    name: Method
    width: 160
  - id: c_000001
    name: Result
    width: 120
rows:
  - id: r_000000
    cells:
      c_000000: Control
      c_000001: 0.42 ± 0.03
  - id: r_000001
    cells:
      c_000000: Treatment A
      c_000001: 0.67 ± 0.04
---
<!-- Generated by Rich Table. Do not edit below — data source is the YAML front-matter above. -->
| Method      | Result       |
| ----------- | ------------ |
| Control     | 0.42 ± 0.03  |
| Treatment A | 0.67 ± 0.04  |
```
````

### Switching themes

Change the `theme:` value (or remove it) and regenerate the pipe table mirror. The visual appearance updates immediately when the plugin re-renders the block.

## Creating a table in a vault note

1. Read the target file with `Read`
2. Identify the insertion point
3. Insert the fenced block using `Edit`

Always generate the pipe table mirror below the `<!-- Generated … -->` comment so the block is valid from the start. The pipe table must:
- Have the same number of columns as `columns[]`
- Have one row per non-hidden data row
- Collapse multi-line cell content to single lines using `<br>`

## Modifying an existing table

1. Read the file and locate the ` ```rich-table ` block
2. Edit **only the YAML front-matter** (above the `<!-- Generated … -->` line)
3. Regenerate the pipe table mirror to match the updated YAML
4. Write back with `Edit`

**Add a row:** append a new entry to `rows[]` with the next sequential `id` (`r_000003`, etc.) and appropriate `cells`.

**Add a column:** append to `columns[]` with the next sequential `id` (`c_000003`, etc.); add the corresponding key to each row's `cells` that needs a value.

**Apply a style:** add or update an entry in `styles[]` using the target reference above.

**Merge cells:** add an entry to `merges[]` with `anchor: rowId.colId` and `end: rowId.colId`. Ensure the anchor cell has the content and covered cells are empty.

**Remove a row/column:** delete the entry from `rows[]` / `columns[]` and remove any references to its ID from `merges[]` and `styles[]`.
